# M6 Multi-Agent Orchestration — Design Document

**Date:** 2026-07-12
**Status:** APPROVED — IMPLEMENTATION READY
**ADR:** M6-ADR-001 — Deterministic Structure + LLM Semantic Enrichment
**Previous milestone:** M5 Product Vertical Slice (v0.1.0)

## 1. Goal

Upgrade SpecFlow from a deterministic single-pipeline Agent Workflow to a
**Controlled Multi-Agent Orchestration System** with:

- Independent specialist agents (Design, Test Strategy, Risk Review)
- Parallel execution of independent agents
- Structured agent-to-agent handoff
- Bounded Review → Revision feedback loop
- Agent-level trace topology
- Single-Agent vs Multi-Agent A/B evaluation

## 2. Non-Goals

- Peer-to-peer agent chat or autonomous negotiation
- Dynamic agent discovery or registration at runtime
- General-purpose DAG scheduler (fixed topology for MVP)
- Replacing or refactoring the existing linear pipeline
- Adding LangGraph, Redis, or external orchestration frameworks

## 3. Architecture: Dual Pipeline Coexistence

```
                    specflow run
                         │
                   ┌─────┴─────┐
                   ▼           ▼
           --mode legacy   --mode multi-agent
           (default)       (new)
                   │           │
                   ▼           ▼
           AgentExecutor   Coordinator
           Analyze         ├── DeterministicPlanner
           Generate        ├── SemanticPlanEnricher
           Review          ├── PlanValidator
                           ├── MultiAgentScheduler
                           └── RevisionController
                   │           │
                   ▼           ▼
              Artifacts     Artifacts
```

**Invariant:** `--mode legacy` uses the existing `AgentExecutor`, `WorkflowEngine`,
`WorkerRegistry`, and three workers. Zero code changes to the legacy path.
All 404 existing tests must continue to pass.

`--mode multi-agent` uses the new `Coordinator` and `AgentRegistry`.
Both modes share `EvidenceCollector`, `ToolRegistry`, `ArtifactStore`, and `TraceRecorder`.

## 4. M6-ADR-001: Deterministic Structure + LLM Semantic Enrichment

### Rule Layer (owns execution authority)

These fields are generated by deterministic code. The LLM MUST NOT modify them:

- Agent set (which agents, how many)
- Dependency graph (who depends on whom)
- Execution stages (compiled from dependencies)
- Token budget per agent
- Tool permissions per agent (always read-only)
- Timeout per agent
- Max revision rounds (hardcoded to 1)
- Required input/output schemas
- Mandatory acceptance criteria (structural gate)

### LLM Layer (owns semantic enrichment only)

The LLM MAY populate these advisory fields:

- `task_description` — what the agent should do
- `analysis_focus` — specific areas to examine
- `evaluation_hints` — supplementary review suggestions (advisory only)
- `repository_scope_hint` — which directories/files to focus on (advisory only)

### Hard Gates

- `evaluation_hints` MUST NOT replace, delete, or weaken mandatory acceptance criteria
- `evaluation_hints` MUST NOT independently determine PASS, REJECT, or Revision
- `repository_scope_hint` MUST NOT expand tool permissions, bypass path rules, or change the repository root
- Effective repository access is determined exclusively by `AgentConstraints.allowed_paths` and tool permissions

### Degradation

```
StructuralDelegationSpec generated
  → LLM enrichment fails
    → Use rule-layer default task descriptions
    → Mark enrichment_status = DEGRADED
    → Multi-Agent Workflow continues execution
```

The LLM Enricher is an enhancement, not a single point of failure.

## 5. Data Models

### 5.1 SchemaRegistry

```python
class SchemaRegistry:
    """Stable, versioned schema registry. Pydantic models are the truth source."""

    def register(self, schema_id: str, model: type[BaseModel]) -> None: ...
    def get(self, schema_id: str) -> type[BaseModel]: ...
    def export_json_schema(self, schema_id: str) -> dict[str, object]: ...
    def list_schemas(self) -> tuple[str, ...]: ...
    def freeze(self) -> None: ...
    @property
    def frozen(self) -> bool: ...
```

**Schema ID format:** `agent/{name}/v{version}/{input|output}`
Example: `agent/design/v1/input`, `agent/design/v1/output`

**Registration rules:**

| Condition | Behavior |
|-----------|----------|
| Same schema_id + same model (object identity: `existing_model is model`) | Idempotent (no-op) |
| Same schema_id + different model | `SchemaConflictError` |
| Non-existent schema_id on `get()` | `SchemaNotFoundError` |
| model is not a `BaseModel` subclass | `ValueError` |
| Registration after `freeze()` | `RegistryFrozenError` |

**Lifecycle:**
App startup → register all schemas → validate AgentIdentity references → freeze() → execute

### 5.3 Structural Plan (Rule Layer)

**`AgentDependency` is the logical truth source for execution order.**
`execution_stages` is compiled from it. `ParallelGroup` is a derived view
(for trace, artifact, visualization) and MUST NOT appear as an independent
input on any structural plan.

```python
@dataclass(frozen=True)
class AgentDependency:
    """Logical truth source for execution order."""
    agent_id: str
    depends_on: frozenset[str]


@dataclass(frozen=True)
class AgentConstraints:
    agent_id: str
    max_execution_seconds: int
    max_token_budget: int
    max_revision_rounds: int         # always 1 for MVP
    allowed_paths: tuple[str, ...]   # empty = entire repo
    denied_paths: tuple[str, ...]


@dataclass(frozen=True)
class RevisionPolicy:
    max_total_rounds: int = 1
    revisable_roles: frozenset[AgentRole] = frozenset({
        AgentRole.DESIGN,
        AgentRole.TEST_STRATEGY,
        AgentRole.RISK_REVIEW,
    })
    final_authority_role: AgentRole = AgentRole.REVIEW
```

**`revisable_roles` uses `AgentRole` (not agent_id)** so that upgrading an agent
implementation (e.g. `design-agent-v1` → `design-agent-v2`) does not require
changing the RevisionPolicy. When constructing a `RevisionTask`, the role is
resolved to a concrete `agent_id` via the `AgentRegistry`.

```python
@dataclass(frozen=True)
class StructuralDelegationSpec:
    """Rule-layer source plan — before compilation. MUST NOT contain compiled fields."""
    plan_id: str
    agents: tuple[AgentIdentity, ...]
    dependencies: tuple[AgentDependency, ...]
    constraints: tuple[AgentConstraints, ...]
    revision_policy: RevisionPolicy
```

```python
@dataclass(frozen=True)
class CompiledStructuralPlan:
    """Compiler output — adds execution_stages and structure_hash."""
    plan_id: str
    agents: tuple[AgentIdentity, ...]
    dependencies: tuple[AgentDependency, ...]
    execution_stages: tuple[tuple[str, ...], ...]  # compiled from dependencies
    constraints: tuple[AgentConstraints, ...]
    revision_policy: RevisionPolicy
    structure_hash: str              # canonical hash of all above
```

**Compilation pipeline:**

```
DeterministicPlanner
  → StructuralDelegationSpec
    → PlanCompiler
      → CompiledStructuralPlan
```

`execution_stages` and `structure_hash` are compiler outputs and MUST NOT
appear on the source `StructuralDelegationSpec`. This makes it impossible
to construct a plan where `execution_stages` contradicts `dependencies`.

**PlanCompiler validation rules:**

- Each agent appears in exactly one stage
- Agents within the same stage have no mutual dependencies
- All dependency agents appear in earlier stages
- Agent sets in tasks and stages are identical

**`ParallelGroup`** is a derived view computed from `execution_stages`:

```python
@dataclass(frozen=True)
class ParallelGroup:
    """Derived view — NOT an independent structural input."""
    group_id: str
    agent_ids: frozenset[str]

    @classmethod
    def from_stages(cls, stages: tuple[tuple[str, ...], ...]) -> tuple["ParallelGroup", ...]:
        """Each stage with >1 agent produces one ParallelGroup."""
        ...
```

It is used in trace output and artifact visualization but is never an
independent field on any structural plan model.

### 5.5 Effective Plan (Merged)

```python
@dataclass(frozen=True)
class AgentTask:
    agent_id: str
    role: AgentRole
    stage: int                        # compiled from execution_stages
    depends_on: frozenset[str]        # from AgentDependency
    constraints: AgentConstraints
    task_brief: SemanticTaskBrief

    @property
    def enriched(self) -> bool:
        """Derived — not an independent field."""
        return self.task_brief.enrichment_status is EnrichmentStatus.ENRICHED


@dataclass(frozen=True)
class EffectiveDelegationPlan:
    plan_id: str
    run_id: str

    structure_hash: str
    semantic_brief_hash: str
    effective_plan_hash: str

    stages: tuple[tuple[str, ...], ...]
    tasks: tuple[AgentTask, ...]
    revision_policy: RevisionPolicy

    generated_at: str

    @property
    def degraded_agents(self) -> tuple[str, ...]:
        """Derived — not an independent field."""
        return tuple(t.agent_id for t in self.tasks if not t.enriched)

    @property
    def enriched(self) -> bool:
        """Derived — not an independent field."""
        return len(self.degraded_agents) == 0
```

**Principle:** Audit fields (`enriched`, `degraded_agents`) are derived from the
single truth source (`SemanticTaskBrief.enrichment_status`). They may be stored
in serialized output for convenience but MUST NOT be independent inputs.

### 5.6 Core Model Dependency Graph

```
AgentDependency          (logical truth source)
       │
       ▼
execution_stages         (compiled, in CompiledStructuralPlan)
       │
       ├── stage numbers  (in AgentTask)

```

```

       │
       ├── AgentTask.enriched          (derived property)
       ├── EffectivePlan.degraded_agents (derived property)
       └── EffectivePlan.enriched      (derived property)
```

## 6. Canonical Hash Rules

### `structure_hash`

Computed on `CompiledStructuralPlan`.

**Participating fields:** agent_id, role, version, prompt_id, prompt_version,
input_schema_id, output_schema_id, tool_permissions,
AgentConstraints (all fields), AgentDependency (all fields),
compiled execution_stages (with agents sorted per stage), RevisionPolicy (all fields)

`description` is excluded — it is display-only metadata.

**Version bump rule:** When an agent's prompt, schema, tool permissions, or core
processing logic changes in a compatibility-affecting way, `AgentIdentity.version`
MUST be incremented. This ensures `structure_hash` reflects the actual execution
contract.

**Excluded:** plan_id, run_id, generated_at, trace IDs, model names

### `semantic_brief_hash`

**Participating fields:** agent_id, task_description, analysis_focus,
evaluation_hints, repository_scope_hint

**Excluded:** enrichment_status, provenance (all fields), generated_at

All fields of `EnrichmentProvenance` are excluded from `semantic_brief_hash`.
Same semantic content generated by a different model or prompt must produce
the same hash; provider, model, prompt, and trace information are audited
separately through `provenance`.

### `effective_plan_hash`

Computed from a canonical envelope:

```json
{
  "hash_version": "specflow-effective-plan-v1",
  "structure_hash": "<structure_hash>",
  "semantic_brief_hash": "<semantic_brief_hash>"
}
```

### Canonical Serialization Rules

- Encoding: UTF-8
- Serialization: Canonical JSON (sorted keys, no trailing commas)
- Object keys: lexicographic order
- Sets (`frozenset`): sorted list before hashing
- Agents within parallel stages: sorted by `agent_id`
- Whitespace: compact (no pretty-print in hash input)
- Algorithm: SHA-256
- Temporal fields: excluded from all plan content hashes

## 8. Plan Compilation Pipeline

```
DeterministicPlanner
        │
        ▼
StructuralDelegationSpec       (rule-layer source, no compiled fields)
        │
        ▼
   PlanCompiler
   ├── Validate DAG (no cycles)
   ├── Compute execution_stages from AgentDependency
   ├── Assign stage number to each agent
   └── Compute structure_hash
        │
        ▼
CompiledStructuralPlan         (immutable, hashed)
        │
        ▼
   SemanticPlanEnricher
   ├── For each agent: call LLM to generate SemanticTaskBrief
   ├── On failure: use degraded_default() with rule-layer descriptions
   └── Compute semantic_brief_hash
        │
        ▼
   PlanValidator                (static plan checks only — no runtime Handoff instances yet)
   ├── Agent set identity check (tasks ↔ stages)
   ├── Stage assignment consistency
   ├── No intra-stage dependencies
   ├── Dependency agents in earlier stages
   ├── Schema ID references exist in SchemaRegistry
   ├── Prompt ID references exist in PromptRegistry
   └── Constraints within legal bounds
        │
        ▼
   EffectiveDelegationPlan (frozen, immutable)
        │
        ▼
   MultiAgentScheduler
        │
        ▼
   HandoffFactory              (creates Handoff when sender completes)
        │
        ▼
   HandoffValidator            (runtime checks — has access to actual payloads)
   ├── source_output_schema_id matches sender AgentIdentity.output_schema_id
   ├── target_input_schema_id matches receiver AgentIdentity.input_schema_id
   ├── Payload validates against target input Pydantic model
   └── payload_ref exists and is readable
        │
        ▼
   Receiver Agent executes
```

## 9. Coordinator Internal Components

```
Coordinator
├── DeterministicPlanner    — generates StructuralDelegationSpec from fixed topology
├── PlanCompiler            — validates DAG, computes execution_stages and structure_hash
├── SemanticPlanEnricher    — calls LLM to fill SemanticTaskBrief per agent
├── PlanValidator           — validates structural consistency and schema references
├── MultiAgentScheduler     — executes stages sequentially, agents within stage in parallel
└── RevisionController      — enforces max 1 round, routes RevisionTask to target agent only
```

## 10. MultiAgentWorkflowState

New state machine, independent of the legacy `WorkflowState`:

```python
class MultiAgentWorkflowState(StrEnum):
    CREATED = "created"
    PLANNING = "planning"            # Coordinator generating DelegationPlan
    ANALYZING = "analyzing"          # RepositoryAnalyst executing
    EXECUTING_SPECIALISTS = "executing_specialists"  # Design/Test/Risk in parallel
    SYNTHESIZING = "synthesizing"    # SynthesisAgent merging
    REVIEWING = "reviewing"          # ReviewAgent judging
    REVISING = "revising"            # Bounded revision (max 1 round)
    COMPLETED = "completed"
    FAILED = "failed"
```

**Transitions:**

```
CREATED → PLANNING → ANALYZING → EXECUTING_SPECIALISTS → SYNTHESIZING → REVIEWING
                                                                         │
                                                   ┌─────────────────────┼──────────────────┐
                                                   ▼                     ▼                  ▼
                                              COMPLETED              REVISING            FAILED
                                                   ▲                     │                  ▲
                                                   │                     ▼                  │
                                                   │              SYNTHESIZING ────────────┘
                                                   │                     │
                                                   └─────────────────────┘
                                                     (after max rounds)
```

Terminal states: `COMPLETED`, `FAILED`.

### Revision Exhausted Terminal Semantics

**`ReviewDecision.REJECT` is a business judgment, not an infrastructure failure.**

```
First Review REJECT
  → Enter REVISING
  → Execute one bounded Revision (target agent → re-Synthesis)
  → Enter REVIEWING

Second Review PASS
  → Enter COMPLETED
  → requires_revision = false
  → revision_exhausted = false

Second Review REJECT
  → Enter COMPLETED  (NOT FAILED)
  → requires_revision = true
  → revision_exhausted = true
```

`FAILED` is reserved for infrastructure errors only:

- Agent throws an unhandled exception
- Schema validation fails and cannot be repaired
- MultiAgentScheduler encounters a fatal error
- Artifact persistence fails
- Illegal state transition
- Hard timeout with no degradation strategy

This is consistent with M4's frozen semantics: business rejection ≠ execution failure.

## 11. Agent Handoff

```python
@dataclass(frozen=True)
class AgentHandoff:
    """Structured handoff between two agents."""

    handoff_id: str
    from_agent_id: str
    to_agent_id: str

    source_output_schema_id: str     # sender's output schema (→ AgentIdentity.output_schema_id)
    target_input_schema_id: str      # receiver's input schema (→ AgentIdentity.input_schema_id)

    payload_ref: str                 # reference to the actual output artifact
    input_hash: str                  # hash of the sending agent's input context
    output_hash: str | None          # None until the receiving agent completes


@dataclass(frozen=True)
class AgentMessage:
    """Structured message from one agent to another."""

    message_id: str
    handoff_id: str                  # links to the AgentHandoff
    sender_agent_id: str
    receiver_agent_id: str
    content_type: str                # "analysis_output" | "design_output" | "revision_task" | ...
    payload_ref: str                 # reference to the actual output artifact
    created_at: str
```

**HandoffValidator checks on handoff construction:**

- `source_output_schema_id` matches `from_agent.AgentIdentity.output_schema_id`
- `target_input_schema_id` matches `to_agent.AgentIdentity.input_schema_id`

For SynthesisAgent (which receives output from three Specialist agents), its
input schema MUST be an explicit aggregate model — not a reuse of any single
Specialist's output schema.

## 12. Multi-Agent Trace Model

Extends the existing `LLMTrace` with agent topology:

```python
@dataclass(frozen=True)
class AgentTraceSpan:
    span_id: str
    agent_id: str
    agent_role: AgentRole
    agent_version: str
    parent_span_id: str              # Coordinator span for all agents; None only for Run root
    handoff_id: str | None           # AgentHandoff that triggered this agent
    stage: int
    stage_started_at: str | None     # ISO timestamp
    agent_submitted_at: str | None
    agent_completed_at: str | None
    stage_completed_at: str | None
    model: str
    latency_ms: int
    input_tokens: int
    output_tokens: int
    status: str
    fallback_level: str | None
    tool_calls: tuple[str, ...]      # tool call IDs
    revision_round: int              # 0 for initial, 1 for revision
    metadata: dict[str, str]

    def as_dict(self) -> dict[str, object]: ...
```

**Trace tree for a complete multi-agent run:**

```
Run (root span, parent_span_id = None)
└── Coordinator span
    ├── RepositoryAnalyst span
    ├── DesignAgent span            ─┐
    ├── TestStrategyAgent span       ├─ parallel (same stage, same parent_span_id)
    ├── RiskReviewAgent span        ─┘
    ├── SynthesisAgent span
    ├── ReviewAgent span
    └── Revision span (if REJECT)
        └── target Agent span
```

Only the Run root span has `parent_span_id = None`. All agent spans are children
of the Coordinator span (or the Revision span for revision-round agents).

## 13. Agent Contracts (per agent)

Each agent registered in `AgentRegistry` must declare:

| Field | Description |
|-------|-------------|
| `agent_id` | Stable, versioned identity |
| `role` | One of `AgentRole` |
| `version` | Semantic version |
| `prompt_id` | System prompt reference (→ PromptRegistry) |
| `prompt_version` | Prompt version for traceability |
| `input_schema_id` | Pydantic model reference (→ SchemaRegistry) |
| `output_schema_id` | Pydantic model reference (→ SchemaRegistry) |
| `context_window` | Which parts of prior outputs this agent sees |
| `tool_permissions` | Which tools this agent may call |
| `token_budget` | Max input + output tokens |

`trace_span_id` is NOT part of the agent contract. It belongs to the execution
context (`AgentExecutionContext`) and is assigned at runtime.

## 14. AgentRegistry

New registry, independent of the legacy `WorkerRegistry`:

```python
class AgentRegistry:
    def register(self, agent: Agent) -> None: ...
    def get(self, agent_id: str) -> Agent: ...
    def get_by_role(self, role: AgentRole) -> tuple[Agent, ...]: ...
    def list_agents(self) -> tuple[AgentIdentity, ...]: ...
```

Unlike `WorkerRegistry` (one worker per role), `AgentRegistry` allows multiple
agents per role (future extensibility), though the MVP fixed topology uses
exactly one agent per role.

## 15. Acceptance Criteria

M6 is complete when:

1. `Coordinator` with 6 internal components runs end-to-end
2. `StructuralDelegationSpec` is generated deterministically from fixed topology; `CompiledStructuralPlan` is produced by PlanCompiler
3. `SemanticTaskBrief` is LLM-enriched; degradation path verified
4. `EffectiveDelegationPlan` is compiled with all 3 hashes
5. 6 agents execute: RepositoryAnalyst, Design, TestStrategy, RiskReview, Synthesis, Review
6. **Design/TestStrategy/RiskReview execute in parallel**, proven at two layers:
   - **Automated test:** controlled agents with submission/completion timestamps — assert all three are submitted before any one completes
   - **Live run trace:** three specialist spans show overlapping time windows, same stage, same parent_span_id, different span_ids
7. Structured `AgentHandoff` with `source_output_schema_id`/`target_input_schema_id` links every agent pair; HandoffValidator checks schema consistency at runtime
8. One bounded Review → Revision → re-Review cycle executes; second REJECT enters COMPLETED with `revision_exhausted=true`
9. Agent-level trace spans form a complete topology tree rooted at Run → Coordinator
10. All 404 legacy tests still pass
11. A/B evaluation framework compares legacy vs multi-agent on the same repo+requirement
12. At least one real repository case produces valid artifacts in multi-agent mode
13. SchemaRegistry with freeze semantics validates all agent schemas
14. `PlanValidator` catches: missing agents, cyclic deps, intra-stage deps, schema mismatch; `HandoffValidator` catches: handoff schema inconsistency, payload validation failure
